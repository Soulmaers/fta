# 📒 Манифест разработки фронтенда

Этот документ фиксирует основные правила и договоренности по написанию кода в проекте. Соблюдение этих пунктов гарантирует чистоту, предсказуемость и легкую поддержку приложения.

### 1. Единообразие имен файлов и компонентов
Имя файла должно в точности повторять название главного компонента внутри. 
(Это позволяет мгновенно находить нужные части кода через поиск.)

### 2. Приоритет async/await над цепочками .then()
Все асинхронные операции описываются через синтаксис `async/await`. 
(Это делает код линейным, читаемым и избавляет от вложенных колбэков.)

### 3. Изоляция бизнес-логики в кастомных хуках
В кастомные хуки выносится **тяжелая логика**: работа с API, сложные расчеты и синхронизация данных. 

*   **В хуках**: Запросы к серверу, валидация данных, сложные цепочки обновлений. Компонент получает от хука уже готовый результат.
*   **В компоненте**: Допускается оставлять **локальную UI-логику**. Например: функции открытия/закрытия модалок, переключение вкладок или простейшие обработчики, которые не затрагивают бизнес-данные приложения. 
(Это упрощает код и избавляет от создания лишних хуков под элементарные задачи.)

### 4. Контроль данных через TypeScript
Все данные должны быть четко описаны (типизированы). Это необходимо, чтобы редактор кода подсказывал нам доступные поля объектов и подсвечивал ошибки и опечатки задолго до того, как мы запустим приложение. 
(Это исключает большинство багов «на лету».)

### 5. Выделенный слой API
Запросы к бэкенду инкапсулируются в отдельных сервисах. Компоненты никогда не делают прямые вызовы. 
(Это позволяет менять эндпоинты или правила в одном месте.)

### 6. Обработка исключений в блоках try/catch
Асинхронные вызовы и сетевые запросы всегда сопровождаются блоками перехвата ошибок. 

### 7. Использование модульных стилей
Стилизация реализуется через изолированные CSS-модули. Использование атрибута `style` (инлайн-стилей) внутри кода нежелательно. Можно использовать только для динамических изменений стилей. 
(Это предотвращает конфликты стилей)

### 8. Иерархия хуков: общие и частные
Логика нужная многим частям приложения, хранятся в общих хуках проекта. Логика, специфичная только для одной формы или страницы, хранится в локальном хуке этого компонента.

### 9. Стандартная навигация роутера
Управление состоянием страницы при смене адреса выполняется через встроенные инструменты роутера (параметры пути). Нежелательно отслеживать изменение URL через «эффекты» для запуска жизненного цикла.
